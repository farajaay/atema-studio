I Executive summary

The codebase is in strong shape. Every previously-patched finding from C-1 through M-11 (plus H-10 and L-8) holds with zero regressions, the mechanical gates are all green, the production bundle is hardened as designed, and the two July P0 migrations correctly closed the bookings PII leak found during album verification.

This pass surfaced one new HIGH finding (H-11): the RLS policy that lets the transfer screen mark a booking awaiting_transfer is row-level, not column-level, so anyone holding the public anon key can blind-overwrite every column — including manage_token, customer_email, total, and event_date — on any booking still in the pending & unpaid window. One dormant MEDIUM (phishing relay in send-whatsapp, inert while WhatsApp is off), one LOW (JSON-LD breakout on admin-authored journal content), and two process gaps (July migrations absent from the CI manifest; the P0 fix untracked in docs/bugs.md) complete the picture.

IV New findings

High  H-11 · Anon UPDATE on bookings is column-unrestricted — takeover / tamper window
Files — migrations-2026-05-rls-hardening.sql:140-149 (policy "Anon update — payment intent only"; mirrored in bundle-existing-db.sql:223) · consumer src/components/BankTransferPayment.tsx:140-142
Defect — RLS is row-level. The policy's USING gates which rows anon may touch (status='pending' and payment_status='unpaid') and WITH CHECK constrains only three columns in the post-image; it does not restrict which columns change. Anyone with the anon key can PATCH /rest/v1/bookings?status=eq.pending&payment_status=eq.unpaid setting any column on all matching rows — overwrite manage_token/album_token (→ self-service page takeover), null total/deposit, move event_date, or corrupt PII. Writes are blind (July lockdown removed anon SELECT), but blind mass-overwrite is enough.
Window — transient per booking (creation → payment-path choice), but always open for whichever bookings are in flight; an attacker can poll.
Fix (~30 min) — drop the anon UPDATE policy; replace the one legitimate write with a single-purpose SECURITY DEFINER RPC (the mark_mood_board_viewed pattern):
drop policy if exists "Anon update — payment intent only" on public.bookings;
create or replace function public.mark_awaiting_transfer(p_booking_id uuid)
returns boolean language sql security definer set search_path = public as $$
  update public.bookings
     set payment_status='awaiting_transfer', payment_method='bank_transfer'
   where id=p_booking_id and status='pending' and payment_status='unpaid'
   returning true;
$$;
grant execute on function public.mark_awaiting_transfer(uuid) to anon, authenticated;
Then switch BankTransferPayment.tsx to supabase.rpc('mark_awaiting_transfer', { p_booking_id: bookingId }).
Medium  M-12 · send-whatsapp has no caller authorisation — latent phishing relay
File — supabase/functions/send-whatsapp/index.ts
Defect — accepts phone, name, total, and manageLink (an arbitrary URL) from the body with no authz beyond the default JWT gate the public anon key satisfies. Anyone could make the studio's WhatsApp number send a legitimate-looking confirmation with an attacker-chosen "إدارة حجزك" link to any phone.
Why not High today — dormant: wa_enabled defaults off (function returns skipped) and Twilio secrets are unset. Live the day WhatsApp is switched on.
Fix — require an internal shared secret (reuse the CRON_SECRET pattern, constant-time compare) passed by create-booking, or accept only a service-role JWT. Gate this into the WA-activation checklist so the switch can't be flipped first. Also: this function speaks Twilio while the rest of the WA stack speaks Meta Cloud API — reconcile when touched.
Low  L-11 · JSON-LD serialisation allows </script> breakout on journal content
File — src/pages/JournalPostPage.tsx:184-204
Defect — post title/excerpt serialised with JSON.stringify into a <script type="application/ld+json"> via dangerouslySetInnerHTML. JSON.stringify doesn't escape /, so a field containing </script> terminates the element and injects markup. Admin-authored only (single trusted admin) → Low, same species as C-1.
Fix (1 line) — JSON.stringify(...).replace(/</g,'\\u003c')
Process  DOC-1 · CI migration manifest frozen at June 2026
File — .github/workflows/supabase-migrations.yml:87-131
Defect — the ordered manifest ends at migrations-2026-06-fix-booking-insert.sql. None of the nine July migrations — including the P0 bookings-pii-lockdown — are listed; the workflow would warn "on disk but not in manifest (skipping)". Currently disabled (owner request), which contains the risk, but it's a trap for whoever re-enables it.
Fix — append the July files in dependency order (lockdown before definer-view; album before album-examples), or add a "frozen / hand-run only" banner.
Process  DOC-2 · P0 PII-lockdown fix untracked in the audit trail
Defect — the July P0 (anon could read the whole bookings table incl. capability tokens) exists only as comments in two migration files: no docs/bugs.md ID, no CLAUDE.md §6 run-list entry, no report. The most severe finding of the July cycle shouldn't live outside the tracker.
